# 2026-07-18：登录后自动把本机 Guest 学习记录归入本人 Learner Space

**状态：** 已采纳；本地 Supabase 已验证，生产部署仍待完成。

## 背景

学生可以在未注册时填写课程信息和完成练习。登录后如果继续使用未隔离的浏览器存储，会失去跨设备恢复能力，也可能在共用电脑上把账号数据暴露给下一位用户；如果要求学生手动逐项导入，又会制造不必要的理解和操作成本。

这里需要区分两件事：把学生当前设备上的记录保存到学生自己的账号，是核心存储行为；允许冰冰、老师、家长或 Agent 读取这些记录，是第三方数据授权。前者不能暗中扩大为后者。

## 选项

1. 登录后仍只使用 localStorage：实现简单，但不具备生产持久性和可靠多租户隔离。
2. 登录后弹出逐项导入确认：控制最显式，但容易产生两个并行档案和遗漏的练习记录。
3. 未登录时只保存到随机 Guest Space；登录后自动、幂等地认领当前设备的 Guest 数据到本人 Learner Space，并清除未隔离的本地副本。

## 决策

采用选项 3。浏览器通过同一个 Supabase 会话解析 `app_user` 与唯一 `learner_space`。课程档案和练习会话只能通过 authenticated RPC 写入；RPC 在单个数据库事务中完成 Guest claim、会话快照和事件后缀追加。`private.guest_space_claims` 保证一个 Guest Space 不能被第二个账号认领，事件 ID 和序号保证网络重试不会重复写入。

匿名状态仍使用本地适配器。登录状态下，云端是唯一持久化来源；网络失败时只在当前页面内暂存并向 UI 返回未持久化状态，不把账号数据降级写回未分区的 localStorage。成功认领后删除当前设备的 Guest 副本。

## 隐私边界

- 自动同步只写入登录学生自己的 Learner Space。
- 邀请码只授予内容 entitlement，不授予任何学习数据读取权。
- 满托、冰冰、老师、家长和 Agent 仍需要学生分别授予有目的、有限时、可撤销的 Grant。
- 生产上线前仍需提供适龄隐私说明、导出、删除、保留期限和第三方授权审计。

## 验证与影响

- `tests/features/learner-data/auth-aware-stores.test.ts` 验证匿名/登录切换、云端失败和共用本地存储边界。
