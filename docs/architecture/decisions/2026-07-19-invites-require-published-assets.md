# 2026-07-19：邀请码只能绑定真实已发布资料

**状态：** 已采纳；数据库、权限与本地真实 HTTP 已验证，运营界面和生产部署仍待完成。

## 背景

权限包的 `active` 只表示包本身仍可使用，不代表包内已经有可交付正文。开发种子中保留 ESAT、TARA、LNAT、UCAT 的未来深度解析包，但对应资源仍是 `draft`。原 `issue_invite` 只检查包是否 active，因此 service-role 可以为只有草稿的包生成邀请码，造成学生注册后没有真实内容可得，与“只承诺已经发布产品”的上位原则冲突。

## 选项

1. 仅由运营人员记住哪些包能发：实现为零，但无法防止误发，也不能由 Agent 独立验证。
2. 只在未来运营页面隐藏草稿包：体验更好，但脚本、REST 或其他集成仍可绕过页面。
3. 在数据库签发函数中建立不可绕过的发布门，同时让页面只读取可签发包。

## 决策

采用选项 3。`issue_invite` 除了验证 package 存在且 active，还要求每个请求包至少绑定一项 `access_tier = entitled`、`publication_status = published` 且具有 `published_at` 的资源。任一包不满足时，整次事务以 `invite_package_unpublished` 失败，不生成任何邀请码。

函数继续只授权 `service_role`，浏览器中的普通学生不能调用；service-role 密钥也不能进入前端。未来冰冰运营台必须调用同一数据库门，不能另写绕过发布状态的签发路径。该门只判断资料可交付性，不授予任何 Learner Space、作答、课程或学习记录读取权。

## 影响与验证

- TMUA 两个已有真实正文的权限包仍可签发；
- ESAT、TARA、LNAT、UCAT 的草稿深度解析包现在无法签发；
- pgTAP 同时证明已发布包成功、草稿包失败和学生角色无签发权限；
- 真实本地 REST 请求使用 service-role 调用草稿包并验证数据库拒绝；
- 运营台、签发审计、撤销 UI 和生产 Supabase 仍是后续工作，不能因本门通过而视为全链路已上线。
