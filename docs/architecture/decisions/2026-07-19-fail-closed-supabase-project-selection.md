# 2026-07-19: Supabase 项目选择先于项目创建

## 背景

满托 100 人封闭 Beta 必须让 staging 与 production 使用两个独立 Supabase 项目。当前账号只有两个名称含糊、状态为 `INACTIVE` 的旧项目；直接复用可能覆盖历史数据，直接创建又可能产生费用。创始人不是技术人员，因此不能把这个选择留给临时命令和 Agent 记忆。

## 选项

1. 直接复用旧项目：速度快，但数据来源、权限、迁移历史和恢复状态未知。
2. Agent 直接创建两个项目：隔离清楚，但可能触发计费，且区域与未成年人数据位置尚需确认。
3. 先只读发现，再用版本化选择计划表达 create、reuse 或 defer：多一步显式决定，但可测试、可审计且不会误写云端。

## 决策

采用选项 3。`production:supabase-inventory` 只读取账号可见项目；`production:supabase-plan` 把 staging/production 的选择转成无副作用计划。两个环境必须指向不同项目；`INACTIVE` 项目在恢复和历史数据审计前不能复用；新项目首轮只能计划 `nano`，并始终标记可能产生费用、需要动作发生前再次批准。计划层没有 create、delete、pause、restore、数据库密码或 service-role 读取路径。

## 影响

- 用户一旦选择 A/B/C，可以把选择写入被 Git 忽略的本地配置，再由同一门禁验证。
- 本决策不创建项目，不代表 Supabase、SMTP、Turnstile、域名或生产恢复已完成。
- 后续执行器必须单独设计精确确认、费用确认、秘密不落 argv/日志、数据库密码安全保存及失败回滚；不得绕过本计划层。
- 若 Beta 规模、数据地区法律意见或负载变化，重新评审区域、实例规格和项目组织归属。
