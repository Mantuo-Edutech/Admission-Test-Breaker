# 站内反馈与题目纠错运行手册

**状态：** 本地链路已验证；生产负责人、告警与 staging 演练待完成
**适用范围：** 题目/解析错误、技术故障、登录权限、隐私安全、功能建议及其他站内问题
**数据边界：** 工单只保存学生主动填写的文字、类别和受限的考试/页面/资源/题号标识，不自动附加答案、练习事件、邮箱、手机号、设备指纹或浏览历史。

## 1. 学生入口与回执

- 在线题目页和提交后的逐题结果都有“报告这道题的问题”入口，自动带入考试、试卷和题号。
- `/feedback` 也可以从账号页直接进入。学生登录后提交并查看本人最近 20 条反馈；RLS 阻止其他学生读取。
- 未登录或账号服务不可用时，页面显示冰冰二维码作为明确的人工后备路径。学生只需发送考试、页面和题号，不应发送密码、作答记录或其他隐私数据。
- 回执格式为 `FB-XXXXXXXX`。同一学生在 24 小时内重复提交完全相同的问题时返回原回执，不新建工单。
- 反馈会进入学生本人 JSON 数据导出；删除账号时随账号级联删除活动数据库中的反馈与审计记录。生产备份中的删除传播仍服从最终备份保留政策。

## 2. 确定性分级

分级由数据库规则完成，不调用 AI，也不消耗 Token。

| 等级 | 自动进入条件 | 测试期处理目标 | 生产要求 |
| --- | --- | --- | --- |
| P1 | 隐私、安全、疑似跨账号越权 | 立即确认风险边界并限制影响面 | 必须配置值班负责人和实时告警 |
| P2 | 题目/解析错误、登录与邀请码问题、核心作答/结果页故障 | 下一个工作日完成初步分诊 | 必须有教研与技术各一名责任人 |
| P3 | 一般页面和非核心功能故障、其他问题 | 三个工作日内分诊 | 每日检查队列 |
| P4 | 功能建议 | 进入产品评审，不承诺实现 | 每周归档和去重 |

这些是封闭测试期内部目标，不应作为对外服务承诺。正式上线前必须由满托明确负责人、备份负责人、工作日定义和升级方式。

## 3. 查看队列

操作人员只能使用受控的 service-role 环境或数据库管理连接，不得把 service-role key 放入浏览器、截图、聊天或前端配置。

```sql
select
  id,
  priority,
  category,
  exam_id,
  resource_id,
  question_id,
  status,
  created_at
from public.student_feedback
where status not in ('resolved', 'closed')
order by
  case priority when 'P1' then 1 when 'P2' then 2 when 'P3' then 3 else 4 end,
  created_at;
```

消息正文只在实际分诊时打开。不要把正文复制到普通协作群；如需协作，使用回执号、类别、题号和最小必要的问题摘要。

## 4. 更新状态

学生不能直接修改状态。操作人员必须通过审计 RPC 更新：

```sql
select public.triage_student_feedback(
  '<feedback uuid>'::uuid,
  'triaged',
  'Assigned to academic review; source comparison pending.'
);
```

允许的目标状态：

- `triaged`：已确认类别和负责人；
- `in_progress`：正在核对或修复；
- `resolved`：修复或解释已经完成，学生可在站内看到；
- `closed`：重复、无法复现或不进入开发；内部备注必须说明理由。

每次更新都会写入 `private.student_feedback_events`。内部备注不对学生显示，但仍按敏感运营数据保护；不要写入不必要的个人信息。

## 5. 题目内容纠错流程

1. 将状态改为 `triaged`，只根据回执读取必要上下文。
2. 找到本地原始资料、来源哈希和当前结构化题目；不能只根据学生文字直接改答案。
3. 两人复核：一名教研核对题干、公式、图形、选项、答案和解析；一名发布者核对资源 ID、题号映射、版本和回归范围。
4. 修改结构化源资产及相应构建/迁移文件，不直接修改构建产物来掩盖来源差异。
5. 执行题库校验、相关页面测试、`pnpm verify:feedback`、`pnpm verify:supabase` 和完整 `pnpm verify`。
6. 按不可变 commit SHA 发布到 staging，人工打开受影响题目和结果页，与原始资料并排复核。
7. 提升同一镜像到 production，确认 `/version.json` 的 commit 与批准版本一致。
8. 将反馈改为 `resolved`，内部备注记录来源版本、变更 commit 和验证结果。

如答案变化会影响已提交结果，不能静默改写历史学习事件。应为内容建立新版本，并在产品界面明确解释旧结果所依据的版本；这一版本化重算能力尚未实现，相关修复必须作为发布阻断项升级。

## 6. 技术与账号故障

- 首先确认影响是否只发生于一个浏览器、一个账号、一个考试或全部学生；不得要求学生发送密码或完整令牌。
- 使用回执中的 route 和 resource/question ID 复现。当前系统不会自动采集 user agent；如浏览器信息确有必要，由学生主动提供最小信息。
- 登录和邀请码问题不得通过手工修改学生学习数据解决。权益变更走正式 invitation/entitlement 流程并留下来源。
- 修复后验证：登录返回原反馈页、课程档案恢复、练习继续、提交和结果页均保持本人隔离。

## 7. P1 隐私与安全升级

发现疑似越权、泄露、service-role 暴露或跨账号读取时：

1. 不在反馈正文中继续索取信息；记录回执号和发现时间。
2. 立即停止可能扩大影响的发布或密钥使用，保留最小审计证据。
3. 通知生产隐私/安全负责人和技术备份负责人；这两个名字与联系渠道在 Beta 前必须填入受控运营系统。
4. 按事故响应流程确认受影响数据、账号、地区、时间范围和恢复动作。
5. 在法律要求的时限内由合格负责人判断通知监管机构和受影响学生的义务。

在正式负责人和告警渠道未配置前，本功能不能把 Beta 的“反馈与支持运营”门禁标为 verified。
